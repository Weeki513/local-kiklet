

# Техническое задание: Local Kiklet (macOS)

## 0) Цель
Собрать локальное macOS‑приложение «Local Kiklet» для записи голоса, офлайн‑транскрибации (локальный Whisper/аналог) и вставки результата в активный инпут (или в буфер обмена, если активного инпута нет). Дополнительно — опциональные действия над текстом через OpenAI API (перевод/стилизация/прочее), включая набор предустановленных действий и возможность добавлять кастомные.

Ключевое: **работоспособность, понятный UX и корректные разрешения macOS**. Визуальный дизайн не критичен.

**Важно:** выбор стека/технологий/конкретной локальной модели — на усмотрение агента. Запрещено требовать от заказчика уточнений по стеку.

## 1) Термины
- **Активный инпут** — текущий фокус ввода текста в любой программе (TextField/TextView/textarea и т.п.).
- **Вставка** — вставить транскрибированный (или обработанный) текст в активный инпут как пользовательский ввод/вставку.
- **Fallback** — если активного инпута нет или вставка невозможна → скопировать текст в буфер обмена и показать уведомление.
- **Действие (Action)** — операция над текстом (например, «Перевести на английский», «Сделать дружелюбнее», «Сжать в 1 абзац», «Список задач»).

## 2) Платформа и поставка
- Целевая ОС: macOS (современные версии; агент выбирает минимальную поддерживаемую).
- Поставка: **установщик для Mac** (например, DMG или pkg) с корректным опытом установки.
- Приложение должно запускаться после установки и проходить первичный онбординг.

## 3) Главные пользовательские сценарии (Must)

### S1. Быстрая запись и вставка
1) Пользователь нажимает горячую клавишу (режим «hold‑to‑record») и удерживает.
2) Идёт запись.
3) Пользователь отпускает клавишу → запись останавливается.
4) Выполняется локальная транскрибация.
5) Результат вставляется в активный инпут.
6) Если активного инпута нет — результат копируется в буфер обмена.
7) Пользователь получает короткую визуальную обратную связь (например, menubar‑индикатор/баннер/notification).

### S2. Переключаемая запись (toggle)
1) Пользователь коротко нажимает горячую клавишу → запись стартует.
2) Повторное короткое нажатие → запись останавливается.
3) Далее как в S1: транскрибация → вставка/буфер.

### S3. Транскрибация + действие
1) Пользователь выбирает действие по умолчанию (например, «Перевести на …»).
2) Запускает запись (S1/S2).
3) После транскрибации выполняется действие.
4) Вставляется **результат действия** (или копируется в буфер), при этом исходная транскрипция может быть доступна в истории.

### S4. Настройка горячих клавиш
1) Пользователь открывает настройки.
2) Переназначает горячие клавиши.
3) Требование: **должны поддерживаться любые клавиши, включая одиночные модификаторы** (например, только ⌥, только ⌘, только ⇧, только Ctrl) — если это возможно в рамках macOS.
4) Горячие клавиши должны работать глобально (поверх любых приложений).

### S5. Онбординг разрешений и API‑ключа
1) При первом запуске приложение проводит пользователя через выдачу необходимых разрешений.
2) Приложение запрашивает API‑ключ OpenAI (для действий над текстом).
3) Приложение должно корректно объяснять, зачем нужны разрешения и что будет работать без них.

## 4) Разрешения macOS (Must)
Приложение должно корректно запросить и получить (в рамках реальных ограничений macOS) разрешения, необходимые для:
1) **Записи аудио с микрофона**.
2) **Глобальных горячих клавиш**.
3) **Определения активного инпута и вставки текста** в сторонние приложения.
4) **Доступа к буферу обмена** (если требуется отдельная обработка).

UX‑требование: онбординг должен показывать статус каждого разрешения (получено/нет) и кнопку/ссылку «Открыть настройки macOS», если пользователь отказал.

Критически важно: в сборке установщика (DMG/pkg) приложение должно **реально** уметь довести пользователя до рабочего состояния без ручных шаманств.

## 5) Функциональные требования

### 5.1 Запись аудио
- Поддержать два режима запуска/остановки: hold‑to‑record и toggle.
- Индикация статуса записи (обязательно): хотя бы минимальная (menubar‑иконка, всплывающее окно, HUD, уведомление).
- Обработка типичных ошибок: микрофон не доступен, нет разрешения, устройство ввода недоступно.

### 5.2 Локальная транскрибация
- Транскрибация выполняется **локально на устройстве**.
- Выбор локальной модели (Whisper или совместимая) — на усмотрение агента.
- Требования к UX:
  - Показ прогресса/статуса (хотя бы «Транскрибирую…»).
  - Возможность отменить текущую транскрибацию.
  - Адекватные таймауты/фейлы с сообщением пользователю.

### 5.3 Вставка текста / буфер обмена
- Если есть активный инпут и вставка возможна → вставить текст туда.
- Если активного инпута нет или вставка не удалась → скопировать текст в буфер обмена.
- В любом случае — показать короткую обратную связь:
  - «Вставлено в …» или «Скопировано в буфер».

### 5.4 Действия над текстом (OpenAI API)
- Приложение должно запросить у пользователя **API‑ключ OpenAI**.
- Действия:
  - Должны быть **предустановленные** (минимум 6–10) и адекватно подобранные под повседневный сценарий.
  - Должна быть возможность **добавлять кастомные действия**.
- Архитектура действий (требования на уровне поведения, не стека):
  - Каждое действие имеет: название, описание, входной формат, целевой язык (если применимо), параметры (если нужны), шаблон/инструкцию выполнения.
  - Действие может работать на:
    - исходной транскрипции,
    - или на выбранном фрагменте из истории.
  - Действие может возвращать:
    - итоговый текст (вставляется/копируется),
    - и метаданные (например, язык, тональность).
- Должен быть выбор действия:
  - действие по умолчанию,
  - быстрое переключение перед записью (например, через меню/палитру).
- Если API‑ключ не задан:
  - транскрибация работает,
  - действия, требующие OpenAI, показывают понятную ошибку и ссылку в настройки.

#### Примеры предустановленных действий (обязательная стартовая пачка)
- «Без обработки (только транскрипция)»
- «Перевести на английский»
- «Перевести на русский»
- «Сделать формально‑деловым стилем»
- «Сделать проще/дружелюбнее»
- «Сжать в 1–2 предложения»
- «Список задач (to‑do)»
- «Краткое резюме»

(Агент может добавить больше, но эти должны быть точно.)

### 5.5 Настройки
Минимально должны быть:
- Горячая клавиша: hold‑to‑record.
- Горячая клавиша: toggle record.
- Переключатель поведения: «вставлять в активный инпут / всегда в буфер / умный режим (по умолчанию)».
- Выбор языка распознавания (auto + ручной выбор).
- Выбор модели/качества (если агент это поддерживает) — хотя бы «быстрее» vs «точнее».
- Выбор действия по умолчанию.
- Управление списком действий (добавить/удалить/дублировать/переименовать/изменить параметры).
- Поле для API‑ключа OpenAI + кнопка «Проверить ключ».

### 5.6 История
- Минимальная история последних N результатов (например, 20–50):
  - исходная транскрипция,
  - результат действия (если применимо),
  - дата/время,
  - имя действия.
- Действия на записи истории:
  - «Скопировать транскрипцию»,
  - «Скопировать результат»,
  - «Повторить действие»,
  - «Удалить запись».

## 6) Нефункциональные требования
- Надёжность: приложение не должно зависать от долгой транскрибации; UI остаётся отзывчивым.
- Безопасность ключа: ключ OpenAI хранится безопасно (на уровне возможностей macOS), не пишется в логи.
- Логи: минимальные диагностические логи (уровни: info/warn/error), в настройках — «экспорт логов».
- Производительность:
  - Приоритет — «достаточно быстро для повседневного использования».
  - Если локальная модель тяжёлая — предусмотреть режим понижения качества.

## 7) UX‑требования (без требований к красоте)
- Приложение должно быть **понятным без инструкции**.
- Минимальный UX‑скелет (допустимые варианты реализации):
  - menubar‑приложение с окном настроек,
  - или обычное приложение + системный трей/меню.
- В любой момент пользователь должен видеть:
  - идёт запись или нет,
  - идёт транскрибация или нет,
  - какое действие выбрано.

## 8) Обработка ошибок (Must)
- Нет разрешения на микрофон → понятный экран/диалог с кнопкой открыть системные настройки.
- Нет разрешения на управление/вставку в другие приложения → объяснение, что будет fallback в буфер.
- Не удалось определить активный инпут → fallback в буфер + уведомление.
- Ошибка локальной модели → понятная ошибка + совет (например, сменить модель/режим, перезапустить).
- Ошибка OpenAI API:
  - неверный ключ,
  - сеть недоступна,
  - лимиты/ошибка API.
  → не ломает транскрибацию, только действие.

## 9) Приёмочные критерии (Definition of Done)
Считаем задачу выполненной, если:
1) Есть установщик (DMG/pkg) для macOS, который ставит приложение.
2) При первом запуске есть онбординг, который:
   - запрашивает/помогает выдать нужные разрешения,
   - запрашивает OpenAI API‑ключ (можно пропустить),
   - доводит до состояния «можно записывать и вставлять текст».
3) Работают оба режима записи (hold и toggle) с глобальными горячими клавишами.
4) Работает локальная транскрибация.
5) Работает вставка в активный инпут; при невозможности — копирование в буфер.
6) Есть предустановленные действия и возможность добавить кастомные.
7) Можно переназначить горячие клавиши, включая одиночные модификаторы (в пределах возможного в macOS).
8) Есть минимальная история.
9) Есть понятные сообщения об ошибках и инструкции по разрешениям.

## 10) Ограничения и свобода агента
- Агент сам выбирает:
  - стек разработки,
  - конкретную локальную модель и способ её поставки,
  - структуру UI.
- Агент обязан обеспечить:
  - локальную транскрибацию,
  - корректные разрешения,
  - установщик,
  - рабочие глобальные хоткеи,
  - вставку/буфер, действия, историю.

## 11) Рекомендация по структуре результата (что агент должен отдать)
- Репозиторий/папка проекта.
- Инструкция сборки (коротко).
- Готовый установщик (DMG/pkg).
- Короткий список: какие разрешения нужны и как приложение их запрашивает.
